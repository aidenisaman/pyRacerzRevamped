# pyRacerzRevamped

## Cycle One Version Description

Cycle One of pyRacerzRevamped focused on three major pillars: internal architecture modernisation, new track content, and a network multiplayer foundation. On the architecture side, the deliverables were a reusable menu event-loop and rendering helpers that eliminated duplicated navigation code across every screen, a polymorphic player input system so bots and human players manage their own controls rather than relying on type checks in the game loop, a broad-phase spatial-grid collision pass to cut O(n²) pair checks down to only nearby cars, hoisted per-frame trigonometry in the car physics to reduce redundant `sin`/`cos`/`sqrt` calls, modernised CLI argument parsing via `argparse`, and binary + zlib-compressed replay serialisation replacing slow string-based frame storage. A reusable `TextInput` helper (with an `IPTextInput` subclass for network addresses) was also introduced, cleaning up all ad-hoc name-entry logic throughout the menus.

The second pillar was new track content. A fully original **Desert** track rework was designed and added, featuring a multi-layer overpass/underpass system where cars visually disappear beneath the bridge and reappear on top — requiring new RGB-channel logic in the checkpoint map (`desertF.png`) and code changes in the game loop to mask cars based on layer. Alongside desert, the **City** track was redisgned and a new **Forest** track was built with their own new features, and a new **Beach** track was added as a further expansion. 

The third pillar was the Phase 1 network multiplayer foundation. A TCP star-topology layer was built with background daemon threads and `queue.Queue` buffers so the pygame loop never blocks on I/O. On top of that, a full lobby system was implemented — host/join selection, live player roster, in-lobby chat, and a networked race where the host broadcasts per-frame car state to spectating clients who render it in real time. The networking layer was also forward-engineered for Phase 2 competitive client racing: `TCP_NODELAY` is already set, application-level sequence numbers drop stale state packets, every car carries a stable `pid`, a server-side player registry maps each connection to its full car info, and a `NetworkClientRace` skeleton class marks the exact implementation target for the next cycle.
